# 给 AI agent 委派任务的统一话术

> 每次委派新 agent 时，用以下话术开头——让 agent 冷启动时**立即知道该读什么、不该碰什么、出问题往哪汇报**。
> 效果远好于自由发挥，已验证。

---

## 标准格式（复制填，[] 内换成你的项目信息）

```text
请先阅读 AGENTS.md（含 STATUS.md 了解当前状态和阻塞项）。
本次任务只处理 [阶段 X / 任务 TY]。
不要修改以下目录和文件：[原始数据路径 / 已有核心代码路径 / ...]
输出落在 [指定输出路径]。
如果发现疑点或新问题，记录到 [指定文档，如 docs/known_issues.md]，不要直接修正或静默覆盖。
本任务已知坑：[列已知风险点，如：某数据集有守恒偏差要标记；坐标系疑似不一致]
交付前必自查：[守恒/边界/格数/量级/单位…，逐条过]
```

### 字段说明

| 字段 | 作用 | 不建议怎么写 | 建议怎么写 |
|---|---|---|---|
| **本任务已知坑** | 把你（主控）已经知道的坑提前告诉 agent，让它别踩 | 空着 | `某数据集 1691 行守恒偏差超 1；单位 g/kg 和 Mg/ha 混用需换算` |
| **交付前必自查** | 做完任务后必须逐条验证的检查项，通过才报完成 | 空着 | `① 栅格行列数=预期；② 输出总和≈输入总和(偏差<0.5%)；③ 单位统一为Mg/ha；④ 无越界/未裁净像元` |

> **为什么这两个字段重要**：我们已验证——同一个 agent，有"必自查"引导的交付比没有的 **缺陷率低得多**。agent 不是不负责，而是不知道你关心哪类错误。把检查项写明确，agent 就能在报完成前自拦截。

## 实际例子（带已知坑 + 自查清单）

```text
# 委派：阶段 3 —— Python 忠实迁移
请先阅读 AGENTS.md（含 STATUS.md 了解当前状态和阻塞项）。
本次任务只处理阶段 3：Python 忠实迁移。
不要修改 data/（原始输入）和 references/（原始参考材料）。
输出落在 outputs/python_legacy/ 和 reports/。
如果发现算法疑点，记录到 docs/known_issues.md，不要直接修正已有逻辑。
如有阻塞，更新 STATUS.md 并在末端标注"阻塞"。
本任务已知坑：(1) 某算法在边界处数值发散，须加防发散保护；(2) 输入数据部分字段含 NA，迁移后要兼容。
交付前必自查：(1) 迁移版输出与原始 R 版输出差异 < 0.1%；(2) 边界 100 行无发散；(3) NA 兼容逻辑通过；(4) scripts/README.md 已更新。
```

## 为什么这比随便说强

| 随便说 | 这个模板 |
|---|---|
| "帮我写个脚本处理一下数据" | agent 不知道管线全貌，可能踩到只读数据、覆盖他人产出、输出丢错目录、疑点无声处理 |
| agent 踩坑后用户再来纠正 | 一次到位——agent 进门就知道边界在哪、往哪输出、出问题往哪记、做完要验什么 |

## 进阶：委派多个 agent 时

在话术末尾加互斥声明：

```text
请先阅读 AGENTS.md（含 STATUS.md 了解当前状态）。
本次任务只处理 [你的任务]。与 [另一个 agent 正在处理的任务] 互不冲突。
如发现交界处需协调，记录到 STATUS.md 并标注"跨 agent 协调"。
```

## 进阶：需要独立复核的任务

对于输出影响论文结论或下游管线的关键任务，在话术末尾加：

```text
交付后安排独立复核：由主控/另一 agent 抽验关键数字（守恒/边界/格数/量级/一致性）。
独立复核通过前，此产物视为"待验"，下游不得启用。
```

## 收工步骤（必做，否则断接）

做完任务后，无论结果如何，必须执行以下收工动作（建议写在话术末尾）：

```text
完事后必须：
1. 更新 AGENTS.md §3（累计快照，带日期）+ §4 看板状态勾选
2. 写 STATUS.md（增量：本轮做了什么/动了哪些文件/踩了什么坑）
3. 如果新增了脚本，更新 scripts/README.md
4. 收工自检：python scripts/check_handoff.py（全过才算交接合格）
5. 如有新数据，更新对应 数据集/来源.txt
```
